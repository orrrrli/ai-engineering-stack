# /sdd-apply [issue#]

Implement one Task issue through the five SDD phases. Never skip a phase. Never implement more than one issue per invocation.

## Prerequisites

```bash
gh auth status   # must succeed with repo + project scope
```

---

## Naming convention (how the backlog encodes the hierarchy)

`prd-to-github-backlog` creates every issue with the title `"<ID>: <title>"`, and the
**ID itself encodes the parent chain** — so you never have to guess relationships:

| Level | ID convention      | Example title              | Labels                       |
|-------|--------------------|----------------------------|------------------------------|
| Epic  | `EPIC-<SLUG>`      | `EPIC-AUTH: Authentication`| `type:epic`                  |
| Story | `<AREA>-<EPIC>-<n>`| `BE-AUTH-1: Login endpoint`| `type:story`, `area:<area>`  |
| Task  | `<story-id>-T<k>`  | `BE-AUTH-1-T1: Add route`  | `type:task`, `area:<area>`   |

Derivation rules:
- **Task → Story:** strip the trailing `-T<k>`. `BE-AUTH-1-T1` → story `BE-AUTH-1`.
- **Story → Epic:** the story's `<EPIC>` segment maps to epic `EPIC-<EPIC>`. `BE-AUTH-1` → `EPIC-AUTH`.
- Tasks and stories also have **real GitHub sub-issue parent links** — use those as the source of truth.

---

## Phase 0 — Resolve the task

**Goal:** Turn whatever the user gave you (raw `#N`, a Task ID like `BE-AUTH-1-T1`, or "list my
tasks") into a concrete Task **issue number** plus its Story and Epic numbers.

Resolve via the **manifest as a cache, verified against the live board** (the chosen strategy):

1. **Read config from the manifest** if present (`manifest.json` at repo root or alongside the
   backlog skill output). Pull `owner`, `repo`, `project_number`, and the `epics[].stories[].tasks[]`
   tree. The manifest IDs let you map a Task ID → Story ID → Epic ID offline.
   - If there is **no manifest**, fall back to pure `gh` label/title queries below.

2. **List tasks** (use when the user says "list my tasks" or gives only an ID/ambiguous input):
   ```bash
   # open tasks on the board, with their IDs in the title
   gh issue list --repo <owner/repo> --label type:task --state open --limit 100
   # or straight from the Project board (shows Status/Sprint columns)
   gh project item-list <project_number> --owner <owner> --format json \
     --jq '.items[] | select(.content.type=="Issue") | {number:.content.number,title:.content.title}'
   ```

3. **Resolve a Task ID → issue number** (exact title prefix match):
   ```bash
   gh issue list --repo <owner/repo> --label type:task --state all \
     --search "in:title <TASK_ID>" --json number,title \
     --jq '.[] | select(.title | startswith("<TASK_ID>:")) | .number'
   ```

4. **Verify against the live board** before proceeding — confirm the manifest-derived parents match
   the real sub-issue links (catches a board edited after generation):
   ```bash
   gh api repos/<owner/repo>/issues/<task#>/parent --jq '.number'   # parent story number
   gh api repos/<owner/repo>/issues/<story#>/parent --jq '.number'  # grandparent epic number
   ```
   If the live links disagree with the manifest, **trust the live board** and note the drift.

**Exit gate:** State the resolved `Task #<n> (<TASK_ID>)`, `Story #<n> (<STORY_ID>)`, and
`Epic #<n> (<EPIC_ID>)`. If the input matched zero or many tasks, list the candidates and ask which
one. Then continue to Phase 1 with these numbers.

---

## Phase 1 — Explore

**Goal:** Understand the task before touching a single file. Use the `Task #`, `Story #`, and
`Epic #` resolved in Phase 0.

1. Fetch the Task issue:
   ```bash
   gh issue view <task#> --repo <owner/repo>
   ```

2. Fetch the parent Story (the sub-issue parent resolved in Phase 0):
   ```bash
   gh issue view <story#> --repo <owner/repo>
   ```
   Read the **acceptance criteria** (Given/When/Then). These are the exit criteria for Phase 4.

3. Fetch the grandparent Epic for broader context:
   ```bash
   gh issue view <epic#> --repo <owner/repo>
   ```

4. Read `.agents/CONTEXT.md` (domain model, business rules, architecture constraints).

5. Locate the relevant code — find the files, components, and layers that this task touches. Report exact file paths and line numbers.

**Exit gate:** State in plain language:
- What this task requires
- Which files will change and why
- Any risks or ambiguities

STOP. Do not continue to Phase 2 until the user confirms the exploration is correct.

---

## Phase 2 — Propose

**Goal:** Define the approach before writing code.

Produce a short implementation plan:
- What interfaces / signatures change (if any)
- Which layer(s) are touched (domain / application / infrastructure / presentation)
- Whether new files are needed
- Whether any dependency is required — if so, name it explicitly and wait for approval
- Any architectural decision that deviates from existing patterns — flag it and ask

**Exit gate:** Present the plan. STOP and wait for explicit user confirmation before writing any code.

---

## Phase 3 — Apply

**Goal:** Implement exactly what was proposed. Nothing more.

Rules:
- Only modify files identified in Phase 2
- Follow all conventions in `~/.claude/CLAUDE.md` and `.agents/CONTEXT.md`
- No new dependencies without asking
- No refactoring outside the task scope — note suggestions after completing
- If you discover something unexpected mid-implementation, STOP and report it

After each file change, state: file path + what changed + why.

---

## Phase 4 — Verify

**Goal:** Prove the task is done against the Story's acceptance criteria.

Checklist:
- [ ] Each Given/When/Then criterion from the Story is met
- [ ] Existing tests still pass (`npm run test` / `dotnet test` / equivalent)
- [ ] New behavior is covered by tests where a correct seam exists
- [ ] No files outside Phase 2 scope were modified

If any criterion is unmet, return to Phase 3. Do not close the issue until all criteria pass.

---

## Phase 5 — Archive

**Goal:** Close the loop on GitHub and capture learnings.

1. Comment on the Task issue with a brief summary of what was implemented:
   ```bash
   gh issue comment <issue#> --repo <owner/repo> --body "Implemented: <one-line summary>. Files changed: <list>."
   ```

2. Close the issue:
   ```bash
   gh issue close <issue#> --repo <owner/repo>
   ```

3. Report done.

---

## Hard rules

- One issue per invocation. If the user mentions multiple, ask which one to start with.
- Never implement a Story or Epic directly — only Tasks. If given a Story/Epic number or ID, use Phase 0 to list that parent's Task sub-issues (`gh api repos/<owner/repo>/issues/<parent#>/sub_issues`) and ask which one.
- If the Task's Story depends on another issue (`depends_on` in the manifest / Story body), check that the dependency is closed before starting Phase 3.
