# /sdd-apply [issue#]

Implement one Task issue through the five SDD phases. Never skip a phase. Never implement more than one issue per invocation.

## Prerequisites

```bash
gh auth status   # must succeed with repo scope
```

---

## Phase 1 — Explore

**Goal:** Understand the task before touching a single file.

1. Fetch the Task issue:
   ```bash
   gh issue view <issue#> --repo <owner/repo>
   ```

2. Fetch the parent Story (listed in the Task body or as a sub-issue parent):
   ```bash
   gh issue view <story#> --repo <owner/repo>
   ```
   Read the **acceptance criteria** (Given/When/Then). These are the exit criteria for Phase 4.

3. Fetch the grandparent Epic for broader context:
   ```bash
   gh issue view <epic#> --repo <owner/repo>
   ```

4. Read `.agents/CONTEXT.md` (domain model, business rules, architecture constraints).

5. Locate the relevant code — find the files, components, and layers that this task touches. Report exact file paths and line numbers.

**Exit gate:** State in plain language:
- What this task requires
- Which files will change and why
- Any risks or ambiguities

STOP. Do not continue to Phase 2 until the user confirms the exploration is correct.

---

## Phase 2 — Propose

**Goal:** Define the approach before writing code.

Produce a short implementation plan:
- What interfaces / signatures change (if any)
- Which layer(s) are touched (domain / application / infrastructure / presentation)
- Whether new files are needed
- Whether any dependency is required — if so, name it explicitly and wait for approval
- Any architectural decision that deviates from existing patterns — flag it and ask

**Exit gate:** Present the plan. STOP and wait for explicit user confirmation before writing any code.

---

## Phase 3 — Apply

**Goal:** Implement exactly what was proposed. Nothing more.

Rules:
- Only modify files identified in Phase 2
- Follow all conventions in `~/.claude/CLAUDE.md` and `.agents/CONTEXT.md`
- No new dependencies without asking
- No refactoring outside the task scope — note suggestions after completing
- If you discover something unexpected mid-implementation, STOP and report it

After each file change, state: file path + what changed + why.

---

## Phase 4 — Verify

**Goal:** Prove the task is done against the Story's acceptance criteria.

Checklist:
- [ ] Each Given/When/Then criterion from the Story is met
- [ ] Existing tests still pass (`npm run test` / `dotnet test` / equivalent)
- [ ] New behavior is covered by tests where a correct seam exists
- [ ] No files outside Phase 2 scope were modified

If any criterion is unmet, return to Phase 3. Do not close the issue until all criteria pass.

---

## Phase 5 — Archive

**Goal:** Close the loop on GitHub and capture learnings.

1. Comment on the Task issue with a brief summary of what was implemented:
   ```bash
   gh issue comment <issue#> --repo <owner/repo> --body "Implemented: <one-line summary>. Files changed: <list>."
   ```

2. Close the issue:
   ```bash
   gh issue close <issue#> --repo <owner/repo>
   ```

3. Self-check: Was there a non-obvious decision, gotcha, or architectural choice made?
   - If yes → suggest: *"This is worth saving to Obsidian — want me to run `/mems`?"*
   - If no → report done.

---

## Hard rules

- One issue per invocation. If the user mentions multiple, ask which one to start with.
- Never implement a Story or Epic directly — only Tasks. If given a Story/Epic number, find its Tasks and ask which one.
- If the Task depends on another issue (`depends_on` in the manifest), check that the dependency is closed before starting Phase 3.
