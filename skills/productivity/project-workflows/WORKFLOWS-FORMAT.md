# WORKFLOWS.md Format

Template for what gets written into `{project}/.claude/WORKFLOWS.md`.

---

```markdown
# {Project Name} — AI Workflows

> Generated by `/project-workflows`. Edit freely — this is yours.
> Stack: {detected stack summary}
> Updated: {date}

## How to use

Invoke any workflow by typing its slash command in Claude Code.
Each entry shows: persona to activate → skill chain to follow → domain context.

---

## Workflows

### /new-{domain-noun}
**Goal:** Spec and build a new {domain noun} end-to-end.

| Step | Tool | What happens |
|------|------|-------------|
| 1 | `/grill-me` | Claude interviews you until the feature is unambiguous |
| 2 | `[requirements-analyst]` | Shapes stories around your domain model |
| 3 | `/to-prd` | Publishes the PRD to your issue tracker |
| 4 | `/to-issues` | Breaks it into independently-grabbable tasks |
| 5 | `[backend-architect]` + `/api-new` | Scaffolds route with validation and auth |
| 6 | `[frontend-architect]` + `/page-new` | Wires the UI |
| 7 | `/lint` | Final pass before opening the PR |

**Domain note:** {one sentence about what makes this specific to the project, e.g. "API routes must use the Prisma client from `lib/db.ts`, not a new instance."}

---

### /audit-{domain-noun}
**Goal:** Harden a {domain noun} before it ships.

| Step | Tool | What happens |
|------|------|-------------|
| 1 | `[security-engineer]` | Reviews route/component for OWASP top-10 |
| 2 | `/api-protect` | Adds auth, rate limiting, input sanitization |
| 3 | `/vibe-security` | Full audit for AI-introduced vulnerabilities |
| 4 | `/api-test` | Generates tests against the hardened API |

**Domain note:** {e.g. "Stripe webhook endpoints must validate the signature before any business logic."}

---

### /ui-{domain-noun}
**Goal:** Verify and polish the {domain noun} UI across breakpoints.

| Step | Tool | What happens |
|------|------|-------------|
| 1 | `[mobile-ui-expert]` | Audits touch targets, overflow, stacking |
| 2 | `/responsive-audit` | Systematic check across breakpoints |
| 3 | `[frontend-architect]` | Applies fixes |
| 4 | `/code-cleanup` | Removes dead Tailwind classes and duplicate logic |

**Domain note:** {e.g. "Client portal pages must work at 320px — clients often use mobile."}

---

### /diagnose-{domain-noun}
**Goal:** Find and fix a hard bug in the {domain noun} with no regressions.

| Step | Tool | What happens |
|------|------|-------------|
| 1 | `/diagnose` | Reproduce → minimise → hypothesise → verify |
| 2 | `/tdd` | Write failing test first, then fix, then refactor |
| 3 | `[refactoring-expert]` | Reviews surrounding code for related smells |
| 4 | `/code-cleanup` | Final pass before the PR |

**Domain note:** {e.g. "Run `npm run seed` before diagnosing data-related bugs — the local DB is often stale."}

---

### /db-{domain-noun}        ← include only if project has a database
**Goal:** Safely evolve the {domain noun} schema.

| Step | Tool | What happens |
|------|------|-------------|
| 1 | `[backend-architect]` | Reviews migration for correctness and reversibility |
| 2 | `/tdd` | Write tests against the new schema before applying |
| 3 | `supabase:types-gen` or `prisma generate` | Regenerate types |
| 4 | `/api-test` | Verify all affected endpoints still pass |

**Domain note:** {e.g. "Always use `prisma migrate dev` in local, never `db push` — we track migration history."}

---

## Personas quick reference

| Persona | When to use |
|---------|------------|
| `[requirements-analyst]` | Before writing any code — clarify the problem |
| `[backend-architect]` | API design, data modelling, business logic |
| `[frontend-architect]` | Components, pages, state, data fetching |
| `[mobile-ui-expert]` | Responsive, touch, mobile-first reviews |
| `[security-engineer]` | Auth, input validation, secrets, OWASP |
| `[refactoring-expert]` | Code smell, tech debt, structural cleanup |
| `[performance-engineer]` | Slow queries, bundle size, render performance |
```
